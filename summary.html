Golden Section (Золотое сечение) — System Summary

What it is: A web-based information system for processing DICOM medical images — specifically, automated cephalometric landmark annotation on 2D/3D dental/orthodontic X-rays. Developed by CodeInside / ЦИФРАЛЕКТ, deployed at gc-stg.codeinside.ru.

Purpose: Uses an ML model to predict ~30 anatomical reference points (Or, Po, S, N, A, B, ANS, PNS, Go, Gn, Me, etc.) on skull/jaw X-rays. Experts review and correct the markup; corrected data feeds back into continuous model retraining.

Users / Roles

  • Super Administrator — full user management: create, edit, activate/deactivate, block/unblock, reset passwords, manage roles, view logs.
  • Administrator — create/edit users, reset passwords, view logs.
  • Doctor (Врач) — search, view, and edit markup on unconfirmed images; view confirmed images.
  • Auditor (Аудитор) — edit markup on both confirmed and unconfirmed images; view action logs.
  • User — baseline role; view and edit own profile.

Main Capabilities

  • Authentication with account lockout after 6 failed login attempts.
  • Personal cabinet with profile (login, full name, phone, city, photo) and password change.
  • User administration — registration, role management, status & password management, activity log.
  • DICOM data ingestion — automatic import from medical storage or manual upload from a local directory.
  • ML-driven predictive markup of cephalometric control points, lines, and angles on 2D/3D DICOM images.
  • Image workflow with versioning and a status pipeline:
    Исходный → В обработке ML → Обработан → Нормализован → Изменен → Подтвержден → Готов к дообучению → Отправлен на дообучение
  • Expert markup editing — add, move, and correct control points; view in anatomical projections; normalize markup.
  • Confirmation workflow — save expert-verified versions and export back to DICOM storage.
  • Continuous ML retraining pipeline: raw → ML prediction → expert → verified → training data → retrained model, with training metrics.
  • Logging — login timestamps and markup-change history, reviewable by administrators and auditors.

Can it replace Diagnocat (e.g. in the dm dental marketplace)?

No — the two systems solve different problems.

The dm backend uses Diagnocat (see backend/internal/services/diagnocat.go and treatment_mapper.go) to:

  • Upload a CBCT (3D DICOM) study and run multi-pathology detection — caries, periodontitis, bone loss, peri-apical lesions, missing teeth, implants, etc.
  • Return structured diagnoses that are mapped to treatment-catalog items to auto-generate a treatment plan grouped by specialty.

Golden Section does something categorically different:

  • Cephalometric landmark annotation — it predicts ~30 orthodontic reference points (S, N, A, B, Go, Gn, Me, …) and lets a doctor correct them.
  • Its output is point coordinates, distances, and angles for orthodontic measurement — not pathology findings.

Gaps that prevent a drop-in replacement

  1. Different clinical domain. Orthodontic cephalometry ≠ general dental pathology diagnosis. Golden Section does not detect caries, perio, lesions, or any of the findings treatment_mapper.go consumes.
  2. No diagnoses / attributes output. dm depends on a structured list of diagnoses to drive treatment-plan generation. Golden Section's documentation describes control points, not pathology codes.
  3. No public API documented. The Diagnocat integration is a REST client (open-session → upload → close → poll analyses → get diagnoses). The Golden Section docs describe a human-in-the-loop web UI and an internal ML retraining pipeline — no API contract is specified for external consumers.
  4. Maturity. The documents are a technical specification plus role manuals for a staging deployment (gc-stg.codeinside.ru) with an in-progress retraining loop — not an SLA-backed commercial service.

A realistic Diagnocat replacement for dm would need to produce CBCT pathology findings in a mappable schema (e.g., Denti.AI, Pearl Second Opinion, or a self-built detection model) and expose an API.
