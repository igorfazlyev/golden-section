2 Назначение и цели создания Системы

2.1 Назначение Системы

Разрабатываемая Система предназначена для обработки рентгенологических снимков челюстей пациентов, в т.ч. нанесения разметки (контрольных точек).

2.2 Цели создания Системы

Целью создания Системы является автоматизация процесса нанесения разметки на DICOM-изображения и определения расстояния между контрольными точками.

2.3 Целевая аудитория

Целевой аудиторией проекта являются медицинские работники в сфере стоматологии.


2.4 Перечень документов, на основании которых создается Система

Рамочный договор № 01/2023 от 29.05.2023 на выполнение работ.

2.5 Ориентировочные сроки начала и окончания работ по созданию Системы

Определяются в соответствии с согласованными сроками проведения работ, определенными в Рамочном договоре № 01/2023 от 29.05.2023 между Заказчиком и Исполнителем.

3 Основные требования к системе

3.1 Требования к системе в целом

Информационная система должна быть реализована, как web-приложение. Доступ пользователей ко всем функциям и ресурсам Системы должен осуществляться через web-интерфейс, с адаптацией под десктоп.

В системе должен быть реализован механизм формирования разметки по контрольным точкам на рентгенологических снимках челюстей пациентов. Реализация предиктивной разметки должна быть основана на технологии машинного обучения.

Система должна обеспечивать возможность просматривать, а также редактировать предварительную разметку снимков.

Медицинские изображения предоставлены в двухмерном (2D) и трехмерном (3D) виде, в формате DICOM.

3.2 Требования к структуре системы

В состав Системы должны входить следующие компоненты:

  • Администрирование;
  • Авторизация;
  • Работа с данными;
  • Работа с медицинскими изображениями;
  • Машинное обучение;
  • Логирование.

Введенные логин и пароль должны пройти процедуру аутентификации. В случае успешной проверки, пользователю предоставляется доступ к Системе. В противном случае, доступ пользователя к Системе должен быть запрещен.

3.2.3 Требования к компоненту «Личный кабинет пользователя»

Компонент системы “Личный кабинет пользователя” должен отображать  следующие данные пользователя:

  • Логин (<часть-имени><часть-фамилии> [Алексей Волков → alvol, Фёдор Бондарчук → fbond, Феликс Бунин → felixb]);
  • ФИО; 
  • Телефон; 
  • Город;
  • Фото.

Пользователь должен иметь возможность просматривать все личные данные, редактировать (изменять) все сведения, кроме логина.

3.2.4 Требования к компоненту «Работа с данными»

Работа c медицинскими изображениями осуществляется по стандарту DICOM.

DICOM поддерживается основными производителями медицинского оборудования и медицинского программного обеспечения. В рамках стандарта охватываются функции создания, хранения, передачи и печати отдельных кадров изображения, серий кадров, информации о пациенте, исследовании, оборудовании, учреждениях, медицинском персонале, производящем обследование и тому подобной информации.

Стандартом DICOM определено два информационных уровня:

  • файловый уровень - DICOM File;
  • сетевой (коммуникационный) - DICOM Network Protocol.

Информационная модель стандарта DICOM для DICOM-файла четырёхступенчатая:

пациент (patient) → исследование (study) → серия (series) → изображение (кадр или серия кадров, image)

Структура DICOM-файла отображена на рисунке 1.

Рисунок 1 - Структура DICOM-файла

  • HEADER - заголовок файла, содержит метаинформацию, в т.ч. информацию о формате файла, версии DICOM, размере файла, идентификаторе исследования и другие метаданные;
  • DATASET - набор элементов данных;
  • Data Element - это основная единица хранения данных, которая содержит информацию о конкретном атрибуте изображения или медицинской информации.

Структура Date Element отображена на рисунке 2.

Рисунок 2 - Структура Data Element

  • Tag - упорядоченная пара 16-битных целых чисел без знака, представляющая номер группы, за которым следует номер элемента. Пример: (0008,001A);
  • VR (value representation) - двухбайтовая символьная строка, содержащая VR элемента данных. Описывает тип данных и формат значения атрибута;
  • Value length - определяет длину значения атрибута;
  • Value Field - содержит данные атрибута.

Компонент «Работа с данными» должен обеспечивать следующие функции по работе с DICOM-изображениями:

  1. Импорт изображений из медицинского хранилища в автоматическом режиме или из локальной директории пользователя вручную
  2. Обработка – нанесение предиктивной разметки по технологии ML
  3. Сохранение предварительно размеченного изображения на контуре ML
  4. Редактирование разметки экспертом и/или подтверждение корректности расположения контрольных точек и линий
  5. Сохранение подтвержденного изображения на контуре ML, для последующего «дообучения» моделей
  6. Экспорт обработанного изображения в медицинское хранилище

DICOM-изображения должны быть версионированы (см. рис. 3), в рамках получения, обработки, сохранения и передачи данных между файловым хранилищем Заказчика и контуром ML, для поддержания согласованности и целостности.

  • Откорректированное изображение экспертом - процесс корректировки описан в пункте 3.2.4 ТЗ

3.2.5 Требования к компоненту «Работа с медицинскими изображениями»

Компонент системы «Работа с медицинскими изображениями» обеспечивает возможность пользователю осуществлять следующие функции:

  1. Поиск подтвержденных или неподтвержденных размеченных снимков в Системе
  2. Просмотр найденных из шага 1 изображений
  3. Внесение коррективов в разметку
  4. Подтверждение изменений, выполненных на шаге 3

3.2.6 Требования к компоненту «Машинное обучение»

Необходимо реализовать функционал по предварительной разметке заданного набора точек (см. Приложение) на DICOM-изображениях челюстей пациентов на основе моделей машинного обучения.

Модели машинного обучения должны быть разработаны по предоставленным размеченным рентгенологическим снимкам пациентов в формате DICOM.

На основе полученных моделей, необходимо реализовать пайплайн непрерывного «дообучения» моделей, применяя экспертную разметку..

Процесс разделяется на 2 этапа:

  1. expert → verified - основной рабочий цикл, в рамках которого данные корректируются для работы основного бизнес-процесса
  2. expert → train data - цикл детальной корректировки разметки некоторых объектов, которые попадут в процесс непрерывного «дообучения»

3.2.7 Требования к компоненту «Логирование»

Система должна фиксировать следующие действия пользователей:

  1. Дата и время входа в Систему;
  2. Изменение разметки снимков.
